# Ouro Next-Token Decoder Self-Rollout S040 Reject

Date: 2026-05-07

## Purpose

The previous in-loop next-token decoder improved some teacher-forced rank
diagnostics but failed greedy generation. This follow-up tested the smallest
exposure-bias fix already available in the training script: train on prefixes
generated by the current model, not only on gold answer prefixes.

Runtime path stayed unchanged:

```text
prompt tokens
-> frozen Qwen donor hidden states
-> QTRM recursive answer-state loop
-> in-loop next-token decoder
-> shared LM head
-> autoregressive text
```

No retrieval, MemoryOS, candidate solver, or side answer channel was used.

## Decision

Reject as a promoted renderer checkpoint.

Self-rollout training was wired correctly, but under a decoder-only trainable
policy it did not repair greedy generation and did not preserve the accepted
halt-gated forced-choice behavior.

## Interpretation

The failure is no longer just a missing self-rollout flag. The answer-state loop
and the tokenizer-level decoder are not being trained as a single causal
answer-generation system. Freezing the accepted core and training only the
decoder creates local token movement but does not make the full recurrent
runtime path better.

Next direction:

```text
stop decoder-only continuation
train answer-state recurrent block + halt gate + decoder jointly under strict
donor-free gates
keep the accepted halt-head checkpoint as the baseline, not as a frozen source
require generation_smoke8 > 0/8 before any promotion
```
